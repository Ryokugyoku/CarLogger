# car-logger-storage

## 目的

CarLoggerの永続化アダプターを提供するクレートです。マスターデータをSQLite、時系列ログをDuckDBへ保存し、アプリケーションが定義するリポジトリ境界を実装します。

## 満たすべき責務

- SQLiteとDuckDBのスキーマ作成、マイグレーション、問い合わせ、トランザクションを管理する。
- 車両、接続履歴、信号定義、CANログ、診断、健康スコア、AI学習データを永続化する。
- マスターデータと大容量ログの保存先を使い分け、統合窓口を提供する。
- 車両単位のデータ分離、論理削除、復元、完全削除を保証する。
- 保持期限、圧縮、チェックポイント、読み取り専用フォールバックを扱う。
- DB行と `domain` / `application` / `health` の型を相互変換する。

## このクレートへ配置するケース

| ケース | 配置先の例 | 判断理由 |
| --- | --- | --- |
| 車両や設定などのマスターデータ | `sqlite.rs`、`vehicles.rs` | 更新整合性と一意制約が中心だから |
| 大量のCAN・PID時系列ログ | `duckdb.rs`、`pid_logs.rs` | 分析・集計向けの列指向保存だから |
| SQLiteとDuckDBをまたぐ操作 | `repository.rs` | 保存先の振り分けと処理順序を隠蔽するため |
| スキーマ追加・既存データ移行 | 対応DBモジュール | 永続化形式の互換性責務だから |
| ログの圧縮・保持期限・削除 | `retention.rs` | 保存容量と履歴意味論の管理だから |
| 診断履歴や健康スコアの保存 | `diagnostics.rs`、`health.rs` | アプリケーションのリポジトリポート実装だから |
| AIモデル世代・推論結果・学習用出力 | `ai.rs`、`learning.rs`、`ai_ui.rs` | 永続化と再現可能な読み出しが中心だから |
| 給油・距離・車両属性候補 | `vehicle_data.rs` | 車両単位の記録整合性とトランザクションが必要だから |

新しいコードは、次のいずれかに該当する場合に配置します。

1. SQL、DB接続、トランザクション、マイグレーションを扱う。
2. 保存形式・索引・クエリ性能・データ寿命に関する処理である。
3. アプリケーションのリポジトリtraitを実装する。
4. DB障害や旧スキーマを安全な読み取り結果へ変換する。

## 配置しないもの

- 保存方式に依存しないデータ型は `car-logger-domain`。
- ユースケースの実行順序やフォールバック方針は `car-logger-application`。
- 健康度の数式やAI特徴量アルゴリズムは `car-logger-health`。
- CANデバイスからの取得は `car-logger-transport`。
- SQL結果の画面表示やGTK状態は `car-logger-gui`。

## データ配置の原則

- SQLite: 車両、設定、接続、信号定義など、参照・更新整合性を重視する小規模データ。
- DuckDB: CANフレーム、PIDサンプル、集計元など、大量追加と分析を重視する時系列データ。
- 複数DB操作では、途中失敗時の再実行可能性と冪等性を設計する。
- 車両IDを持つデータは、問い合わせ・削除・集計のすべてで車両境界を維持する。

## 変更時の確認

- 新規DBと既存DBの両方でマイグレーションをテストする。
- マイグレーションは再実行しても結果が変わらないことを確認する。
- 読み取り専用、部分的な旧スキーマ、削除、復元、車両間分離をテストする。
- 生ログを破壊する処理には、保持意味論と境界件数のテストを追加する。
