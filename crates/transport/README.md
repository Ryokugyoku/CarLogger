# car-logger-transport

## 目的

車両や記録済みデータからCANフレームを取得し、`car-logger-application` が定義する入力ポートへ適合させるクレートです。OS、デバイス、プロトコル固有の差異をこの境界内へ閉じ込めます。

## 満たすべき責務

- Serial/ELM327、Linux SocketCAN、ログリプレイを共通のCAN入力として提供する。
- 接続可能なインターフェースを列挙する。
- デバイスの初期化、読み取り、タイムアウト、応答解析を行う。
- OBD-II要求・応答、VIN、対応PID、DTCなど通信プロトコル固有の変換を行う。
- 外部ライブラリの型とエラーを、アプリケーション境界の型へ変換する。

## このクレートへ配置するケース

| ケース | 配置先の例 | 判断理由 |
| --- | --- | --- |
| 新しいCANアダプター対応 | `serial.rs` または新規モジュール | 物理・論理通信方式に固有だから |
| Linux CANインターフェース対応 | `socketcan_transport.rs` | OS APIに依存するため |
| 保存ログを入力として再生 | `replay.rs` | `CanFrameSource` の入力アダプターだから |
| ELM327の初期化コマンドや応答解析 | Serialプロトコル用モジュール | デバイスプロトコル固有だから |
| OBD-IIの複数ECU応答やDTC解析 | Serial/OBD解析コード | ワイヤ形式からアプリケーション型への変換だから |
| ポート・インターフェース探索 | `list_connected_interfaces` | OS上の通信資源を調べる処理だから |

新しいコードは、次を満たす場合に配置します。

1. 入出力先が車両、CANデバイス、Serialポート、SocketCAN、または入力リプレイである。
2. 主な関心がバイト列、コマンド、フレーム境界、タイムアウト、デバイス能力である。
3. 結果を `domain` または `application` の型へ変換できる。

## 配置しないもの

- 接続再試行回数やユーザーキャンセル方針は `car-logger-application`。
- CANフレーム自体の共通定義は `car-logger-domain`。
- 受信フレームの保存、SQL、保持期限処理は `car-logger-storage`。
- センサー値から健康度を求める処理は `car-logger-health`。
- 接続ダイアログやエラー表示は `car-logger-gui`。

## 実装ルール

- ブロッキングI/Oには必ずタイムアウトまたはキャンセル可能な境界を持たせる。
- 不完全応答、複数行応答、大文字小文字、複数ECU、範囲外値を安全に扱う。
- Linux限定実装は `cfg(target_os = "linux")` で隔離する。
- パーサーは可能な限り純粋関数にし、実デバイスなしでテストする。
- デバイス固有型を公開APIへ露出しない。

## 変更時の確認

- 正常応答、不完全応答、`NO DATA`、複数フレーム、オーバーフロー境界をテストする。
- 優先度付きPIDポーリングの頻度を壊していないことを確認する。
- 対象OS固有コードを変更した場合は、条件付きコンパイルも確認する。
