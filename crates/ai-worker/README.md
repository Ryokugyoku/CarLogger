# car-logger-ai-worker

## 目的

Rustアプリケーションと分離されたPython/TensorFlowワーカーのライフサイクルおよび通信を管理するクレートです。重いAIランタイムの障害や遅延を、CANロギング本体から隔離します。

## 満たすべき責務

- Python実行環境とワーカースクリプトを安全な優先順位で検出する。
- ワーカープロセスを起動し、標準入出力のJSONプロトコルで通信する。
- プロトコルバージョンとヘルスチェック結果を検証する。
- 要求ID、タイムアウト、終了、クラッシュを型付きエラーとして扱う。
- AI推論を非同期キューへ渡し、ロギング処理をブロックしない。
- キュー容量を有界にし、クラッシュループをセッション単位で抑止する。
- 強制終了と通常終了を冪等かつ安全に行う。

## このクレートへ配置するケース

| ケース | 配置例 | 判断理由 |
| --- | --- | --- |
| Python環境・スクリプト探索 | `WorkerLaunch` | 外部AIランタイムの起動責務だから |
| JSON要求・応答の型追加 | `Request`、`Response` | Rust/Python間プロトコルだから |
| ワーカーヘルスチェック | `spawn_verified` | 利用前のプロセス互換性確認だから |
| 非同期推論キュー | `AsyncInferenceWorker` | 重い推論から呼び出し元を隔離するため |
| タイムアウト・クラッシュ・終了処理 | `Worker`、`WorkerError` | 子プロセス管理の責務だから |
| 連続クラッシュ抑止 | `CrashGuard` | AIプロセス障害から本体資源を守るため |

新しいコードは、次のいずれかに該当する場合に配置します。

1. Python/TensorFlowプロセスの探索、起動、監視、終了を扱う。
2. RustとAIワーカー間の通信プロトコルを扱う。
3. AI処理を非同期化・有界化し、本体から障害隔離する。
4. AIランタイムの診断情報や互換性を検証する。

## 配置しないもの

- 特徴量生成、スコア校正、健康判定は `car-logger-health`。
- モデル世代、推論結果、学習ジョブのDB保存は `car-logger-storage`。
- AI失敗時の統計フォールバック方針は `car-logger-application`。
- AI状態の画面表示は `car-logger-gui`。
- Python側のモデル実装は `python/ai_worker` 配下で管理する。

## 実装ルール

- 呼び出し元スレッドでTensorFlow完了を待たない。
- すべての要求に一意なIDとプロトコルバージョンを付ける。
- 応答ID・種類・バージョンの不一致を正常結果として扱わない。
- 子プロセスと読み取りスレッドをリークさせない。
- 環境変数指定は自動探索より優先し、存在しないパスは明示的に失敗させる。

## 変更時の確認

- 正常応答、タイムアウト、クラッシュ、プロトコル不一致、強制終了をテストする。
- キュー満杯でもロギング側が待たされないことを確認する。
- Python環境がない開発環境でも、探索テストが安全に完了するようにする。
