# car-logger-health

## 目的

車両信号から健康状態を評価するための、保存方式やUIに依存しない計算クレートです。統計評価、セッション推定、特徴量生成、AI結果との統合を再現可能な純粋ロジックとして提供します。

## 満たすべき責務

- 標準PIDを健康評価用の正規化された信号へ変換する。
- 走行セッション、走行状態、データ品質、学習状態を推定する。
- median、MADなどのロバスト統計と相対スコアを計算する。
- 複数コンポーネントのスコア、信頼度、カバレッジを統合する。
- AI学習・推論向けの再サンプリング、正規化、信号選択、ウィンドウ生成を行う。
- AI推論結果の校正、セッション評価、統計結果との統合、通知抑制を行う。
- アルゴリズムと特徴量スキーマのバージョンを明示する。

## このクレートへ配置するケース

| ケース | 配置先の例 | 判断理由 |
| --- | --- | --- |
| PID値から健康評価用サンプルへの変換 | `lib.rs` | 保存・表示に依存しない計算だから |
| 中央値、MAD、重み付き集計 | `lib.rs` | 決定的な統計アルゴリズムだから |
| 走行状態やセッション境界の推定 | `lib.rs` | 健康評価モデルの意味論だから |
| 1秒再サンプリングや欠損マスク | `ai_features.rs` | AIへ渡す特徴量契約だから |
| 学習・推論ウィンドウ生成 | `ai_features.rs` | 再現可能なモデル入力生成だから |
| AIスコアの校正と全体状態への統合 | `ai_condition.rs` | 健康判定アルゴリズムだから |
| 通知の継続・回復判定 | `ai_condition.rs` | 単なるGUI通知ではなく健康状態の時系列規則だから |

新しいコードは、次を満たす場合に配置します。

1. 同じ入力に対して同じ結果を返す計算として実装できる。
2. DB、ファイル、ネットワーク、GTK、Pythonプロセスを必要としない。
3. 車両健康度、特徴量、AI評価の意味論に直接関係する。
4. アルゴリズムまたはスキーマのバージョン管理対象になり得る。

## 配置しないもの

- 生の車両・CAN共通型は `car-logger-domain`。
- 健康スコアの保存・検索・バックフィル進捗は `car-logger-storage` と `car-logger-application`。
- TensorFlowワーカーの起動と通信は `car-logger-ai-worker`。
- CAN/OBD通信は `car-logger-transport`。
- グラフ描画、色、翻訳、画面用文言は `car-logger-gui`。

## アルゴリズム変更のルール

- NaN、無限値、欠損、ゼロ分散、時刻逆転を安全に処理する。
- 将来の再計算結果が変わる変更では、アルゴリズムまたは特徴量スキーマのバージョン更新を検討する。
- 閾値は名前付き定数とし、由来と単位を明確にする。
- 学習データと評価データの時間的リークを発生させない。
- サンプル数に比例してメモリが無制限に増える処理を避ける。

## 変更時の確認

- 境界値、欠損、外れ値、ゼロMAD、低カバレッジをテストする。
- ウィンドウ生成は順序と内容の再現性をテストする。
- AI利用不能時も統計評価が成立することを確認する。
