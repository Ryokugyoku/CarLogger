# car-logger-application

## 目的

CarLoggerのユースケース、処理手順、外部機能との境界を定義するクレートです。ドメイン型を使って「何を行うか」を表現し、通信方式、DB、GUIの具体実装から切り離します。

## 満たすべき責務

- CAN入力・保存・診断・健康スコアなどのポートをtraitとして定義する。
- 接続、再接続、車両識別、PID探索、フォールバックなどのユースケースを調整する。
- リアルタイム表示に必要な最新状態を、安全かつ有界に管理する。
- 車両ダッシュボード向けの期間計算、走行距離、給油、集計規則を提供する。
- 外部実装の失敗をアプリケーション上の結果へ変換し、GUIへ技術詳細を漏らさない。

## このクレートへ配置するケース

| ケース | 配置先の例 | 判断理由 |
| --- | --- | --- |
| 外部入力や保存先に要求する能力 | `CanFrameSource`、`CanFrameRepository` | 実装ではなくユースケース側が必要な契約だから |
| 接続の再試行やキャンセル | `connection.rs` | 複数の通信実装に共通する処理方針だから |
| 安全なPID探索の進行管理 | `pid_scan.rs` | 通信コマンドそのものではなく、探索手順だから |
| 信号式の検証・評価 | `pid_formula.rs` | 保存前・表示時など複数境界から利用されるアプリケーション規則だから |
| CAN IDごとの最新状態 | `realtime.rs` | 並行状態管理は純粋なドメイン型ではなく実行時サービスだから |
| 期間別の車両費用集計 | `vehicle_dashboard.rs` | DB問い合わせやGTK描画ではないユースケース計算だから |
| AIエンジン失敗時の統計フォールバック | `ScoreEngine`、`infer_with_fallback` | エンジン選択という処理方針だから |

新しいコードは、次のいずれかに該当する場合に配置します。

1. 複数のインフラ実装を同じ手順で利用する。
2. ユーザー操作を、ドメイン処理と外部I/Oの組み合わせへ変換する。
3. GUI以外からも実行できる処理フローである。
4. 外部クレートに実装させるポートを定義する。
5. 実行中だけ保持する有界なアプリケーション状態である。

## 配置しないもの

- CarLogger共通のデータ語彙だけなら `car-logger-domain`。
- ELM327コマンド、Serial読み取り、SocketCAN APIは `car-logger-transport`。
- SQL、トランザクション、DB固有の行変換は `car-logger-storage`。
- 統計式、健康スコアアルゴリズム、AI特徴量は `car-logger-health`。
- Pythonプロセス起動やJSONLプロトコルは `car-logger-ai-worker`。
- GTKイベント、ウィジェット更新、表示文言は `car-logger-gui`。

## 設計上のルール

- ユースケースは可能な限りtraitへ依存し、具象DBや通信実装を受け取らない。
- 長時間処理にはキャンセル、進捗、タイムアウトの境界を設ける。
- リアルタイム処理では履歴を無制限に保持しない。
- 純粋計算はI/Oから分離し、単体テスト可能にする。
- 新しい責務が既存モジュールと独立して成長する場合は、まず専用モジュールへ分け、独立利用・依存削減の効果が明確な場合にクレート分離を検討する。

## 変更時の確認

- 正常系だけでなく、キャンセル、タイムアウト、部分失敗、フォールバックをテストする。
- ポート変更時は `transport` と `storage` の実装、およびGUIの呼び出し側を確認する。
- 並行状態は競合更新と上限・クリア動作をテストする。
