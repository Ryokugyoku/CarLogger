# car-logger-domain

## 目的

CarLoggerで扱う中核データを、GUI、DB、OS、通信方式から独立した語彙として定義するクレートです。依存方向の最も内側に位置し、他クレート都合の処理を持ち込みません。

## 満たすべき責務

- 車両、CANフレーム、信号定義など、システム全体で共有するドメイン型を定義する。
- ID、列挙値、値オブジェクトの意味と不変条件を表現する。
- 永続化形式や通信ライブラリ固有型に依存しない、シリアライズ可能なデータ契約を提供する。
- 時刻、単位、識別子の意味を全利用側で統一する。

## このクレートへ配置するケース

| ケース | 配置例 | 判断理由 |
| --- | --- | --- |
| 複数の上位クレートが同じ意味で使うデータ | `Vehicle`、`CanFrame` | 特定ユースケースや技術に依存しない共通語彙であるため |
| 業務上の種類を表す列挙 | `FuelType`、`SignalKind` | DBの文字列やGUIの表示名ではなく、業務上の有限集合であるため |
| 外部形式から切り離した観測結果 | `CanIdObservation`、`DecodedSignalValue` | SocketCAN、Serial、DuckDBなどの型を露出しないため |
| 全層で同じ意味を持つ識別子 | `VehicleId`、`CanId` | 層ごとの型解釈のずれを防ぐため |

新しいコードは、次をすべて満たす場合に配置します。

1. GUIがなくても意味が成立する。
2. SQLite、DuckDB、Serial、SocketCANなどを知らなくてよい。
3. スレッド、ロック、ファイル、ネットワークなどの実行制御を行わない。
4. 特定の画面や単一ユースケースだけでなく、CarLoggerの共通語彙として説明できる。

## 配置しないもの

- 接続、再試行、PID探索などの処理手順は `car-logger-application`。
- Serial、SocketCAN、リプレイ入力は `car-logger-transport`。
- SQL、マイグレーション、保存モデルは `car-logger-storage`。
- 健康度の統計計算やAI特徴量生成は `car-logger-health`。
- `DashMap`、`Mutex`、バックグラウンドスレッドなどの状態管理は上位クレート。
- GTKウィジェット、翻訳文言、画面用フォーマットは `car-logger-gui`。

## 変更時の確認

- 公開型のフィールド変更は、保存データやシリアライズ互換性への影響を確認する。
- 技術依存のクレートを新規依存に追加しない。
- 値の検証規則を追加する場合は、境界値と不正値の単体テストを追加する。
