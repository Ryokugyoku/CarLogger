# car-logger-gui

## 目的

CarLoggerのデスクトップ実行ファイルとGTKプレゼンテーション層を提供するクレートです。ユーザー操作、画面状態、バックグラウンド処理の起動を組み立て、下位クレートの機能を利用可能なUIとして提示します。

## 満たすべき責務

- アプリケーション起動、依存オブジェクトの生成、画面遷移を構成する。
- GTKウィジェット、イベントハンドラ、表示状態、入力検証結果を管理する。
- 接続、リアルタイムロギング、健康評価、AI状態、設定を画面へ結び付ける。
- 長時間処理をバックグラウンドで実行し、UIスレッドへイベントを安全に返す。
- 翻訳、ロケール、表示フォーマット、グラフ、サイドバーを提供する。
- 更新確認、更新ファイル検証、再起動時のUI状態復元を扱う。
- DBや通信の技術的エラーを、操作可能なユーザー向け状態へ変換する。

## このクレートへ配置するケース

| ケース | 配置先の例 | 判断理由 |
| --- | --- | --- |
| アプリ起動と依存の組み立て | `main.rs` | composition rootとして具象実装を選ぶため |
| リアルタイム画面更新 | `live_dashboard.rs` | GTKウィジェットへの反映だから |
| ロギングスレッドとUIイベントの橋渡し | `realtime_logging.rs` | デスクトップ実行時のオーケストレーションだから |
| 健康度、AI、ロググラフ画面 | `ui/health.rs`、`ui/ai_condition.rs`、`ui/log_charts.rs` | 表示とユーザー操作に固有だから |
| CAN ID編集画面 | `ui/can_id_manager.rs` | フォーム、一覧、選択状態を管理するため |
| 設定、言語、翻訳 | `config.rs`、`localization.rs`、`ui/settings.rs` | デスクトップ表示環境に固有だから |
| 自動更新とUI状態復元 | `updater.rs` | 配布済みGUIアプリのライフサイクルだから |
| 色、ラベル、単位、レイアウト | 各UIモジュール | プレゼンテーション上の判断だから |

新しいコードは、次のいずれかに該当する場合に配置します。

1. GTK型、ウィジェットID、ユーザーイベント、画面状態を扱う。
2. 下位クレートの結果を表示形式へ変換する。
3. デスクトップアプリとして具象リポジトリや通信アダプターを組み立てる。
4. UIスレッドとバックグラウンド処理の間を橋渡しする。
5. ローカライズ、アクセシビリティ、更新、画面状態復元に関係する。

## 配置しないもの

- GTKなしで成立する業務型は `car-logger-domain`。
- 接続・探索・集計など再利用可能なユースケースは `car-logger-application`。
- SerialやSocketCANのバイト処理は `car-logger-transport`。
- SQLやDB行変換は `car-logger-storage`。
- 健康スコアやAI特徴量の数式は `car-logger-health`。
- Pythonワーカーのプロセス管理は `car-logger-ai-worker`。

## 実装ルール

- UIスレッドで通信、DB全件走査、AI推論などの長時間処理を行わない。
- バックグラウンド結果には世代や対象IDを持たせ、古い結果で現在画面を上書きしない。
- ウィジェットから直接SQLを発行せず、application/storageの公開境界を利用する。
- 画面に表示する一覧と時系列点数には上限を設ける。
- ユーザー向け文字列は翻訳可能にし、技術エラーをそのまま露出しない。
- `main.rs` が肥大化する変更は、画面または機能単位のモジュールへ分離する。

## 変更時の確認

- 表示ロジックはGTK初期化を必要としない純粋関数へ切り出して単体テストする。
- 非同期結果の順序逆転、切断、キャンセル、画面破棄後の応答を確認する。
- 翻訳キー、ウィジェットID、対象OSごとの更新処理を確認する。
